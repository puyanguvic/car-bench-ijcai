\begin{abstract}
Tool agents often let one language model both interpret intent and authorize
external effects.  We instead present PACT, a contract-guided Track~2 agent
that treats its Cerebras-hosted model as an \emph{untrusted semantic
compiler}.  The model emits a compact strict-schema wire plan; a deterministic
kernel then performs strict decoding, typed-DAG validation, live Draft~2020--12
JSON Schema checking, and evidence-aware execution.  PACT exposes at most one
outstanding action, recompiles at semantic data boundaries, scopes confirmation
to an exact operation--argument--capability-snapshot triple, and carries
successful action obligations into the evidence conditions of a typed terminal
outcome.  One
verification-guided repair is permitted; plans containing an \textsc{Act}
undergo one separate second-pass stochastic policy audit and deterministic
reverification.  Provider retries obey finite retry and cumulative-wait
budgets, while final A2A text responses report mutually exclusive visible
completion and thinking-token metrics.  Guarantees cover local structure,
contracts, transitions, and evidence---not semantic interpretation.  The
release suite exceeds 350 tests; a historical public smoke trajectory is
reported only as engineering evidence, not a performance estimate.
\end{abstract}

\section{Introduction}

CAR-bench evaluates whether tool agents act consistently and respect limits
under ambiguous or incomplete information \cite{kirmayr2026car}.  Through the
Agent2Agent (A2A) protocol \cite{a2a2026}, the evaluator supplies policy text,
messages, live tool declarations, and tool results; it alone executes external
operations.  Track~2 additionally rewards useful behavior under a bounded
Cerebras inference route.  A ReAct-style loop \cite{yao2023react} is flexible,
but the same stochastic component commonly interprets the request, constructs
arguments, decides that preconditions hold, and declares success.

PACT separates those responsibilities.  The LLM translates language into a
small obligation program, but has no execution authority.  A local monitor
admits the program only after checking its syntax, graph, current capability
contracts, resource bounds, confirmations, and evidence dependencies.  This
resembles proof-carrying code's separation of an untrusted producer from a
small checker \cite{necula1997pcc} and runtime verification's monitoring of
concrete executions \cite{leucker2009runtime}.  This is an architectural
analogy, not a claim that natural-language meaning is formally proved: the
accepted artifact is only a locally checkable structural/provenance
certificate.

Our contributions are: (i) a provider-compatible, strict wire format that
decouples structured generation from dynamic tool schemas; (ii) a typed PlanIR
and receding-horizon obligation runtime with evidence-gated typed outcomes,
individually consumed confirmation scopes, and contract revalidation at
emission; and (iii) a precise assurance boundary plus reproducible
failure-oriented tests and compute telemetry.  The trusted kernel is
domain-agnostic: it contains no task IDs, benchmark utterances, or automotive
workflow branches.

\paragraph{Positioning.}
PACT is neither a direct model-to-tool interface nor a catalog of
benchmark-shaped workflows.  Direct tool calling constrains the surface form
of an action, but syntactically valid JSON does not establish that the named
operation is still available, its arguments satisfy a changing recursive
schema, a confirmation applies to this exact effect, or a result supports the
claimed terminal outcome.  Empirical work on structured generation similarly
finds a gap between nominal schema conformance and reliable generation across
real schemas \cite{geng2025structured}.  PACT therefore uses two languages: a
small provider-facing wire language optimized for constrained decoding, and a
richer local PlanIR checked against the concrete runtime state.  Conversely,
unlike a hand-authored finite-state policy, the PlanIR is synthesized anew from
the current policy, request, evidence, and capabilities.  Selectivity occurs at
each verified plan horizon: the model proposes semantics, while the kernel owns
only transitions that are structurally determined by the accepted graph.

\section{PACT Architecture}

\subsection{State and Untrusted Compilation}

For one active goal in A2A context $q$, PACT maintains
\begin{equation}
 S_t^q=(P,g,H_t,T_t,E_t,G_t,p_t,O_t,C_t,M_t),
 \label{eq:state}
\end{equation}
where $P$ is the evaluator-supplied policy, $g$ the current goal, $H_t$ the
bounded compiler-visible suffix of a longer context conversation, $T_t$ the
live tool contracts, $E_t$ a per-goal append-only evidence ledger, $G_t$ the
accepted plan, $p_t$ its single pending action, $O_t$ carried completion
obligations, $C_t$ consumed confirmation scopes, and $M_t$ accumulated
inference metrics.  Each exact scope in $C_t$ is projected out of future
compiler inputs after use.  Context cancellation deletes this state while
retaining a stable per-context lock to serialize concurrent turns.

The semantic compiler $C_\theta$ receives $P$ in a separate trusted system
partition and the goal, recent events, $T_t$, and a typed projection of $E_t$
as task state.  It proposes a wire plan
\begin{equation}
 W_t=C_\theta(P,g,H_t,T_t,\operatorname{view}_{C_t}(E_t),O_t).
 \label{eq:compile}
\end{equation}
The provider schema is strict: every object forbids extra properties, and its
canonical serialization is capped at 5,000 characters to match the
Cerebras \emph{response-schema} contract \cite{cerebras2026structured}; this is
not the generated candidate limit, which is independently 64,000 characters.
Dynamic
arguments cannot be represented as an unrestricted object under that contract,
so each operation node carries an \texttt{arguments\_json} string.  Locally,
PACT rejects duplicate keys and non-finite numbers, decodes that string to one
JSON object, and validates it against the actual live schema.  Thus provider
constrained decoding is an early filter, not the authority.  If local checking
rejects the first proposal, one repair prompt contains value-redacted typed
issues; a second rejection terminates safely.  If the verified candidate
contains any \textsc{Act}, a separate second-pass model call reviews the complete plan
against policy, goal, evidence, capabilities, and argument provenance.  Its
full corrected candidate is decoded and verified again; failure rejects the
plan.  This audit is a stochastic cross-check, not a semantic proof, and is
skipped for action-free plans.  The submitted default is
\texttt{gpt-oss-120b}, medium reasoning, and an 8,192-token output budget.  A
provider \texttt{length} finish is
recorded as typed truncation and is not repaired by repeating the same call;
empty content is likewise a typed invalid candidate and never reaches PlanIR.

\subsection{Typed PlanIR and Live Verification}

After decoding, a plan is a bounded directed acyclic graph with nodes from
\[
 \mathcal N=\{\textsc{Observe},\textsc{Ask},\textsc{Confirm},
              \textsc{Act},\textsc{Respond}\}.
\]
Node IDs and evidence keys are unique; dependencies must exist and be acyclic;
every node must reach exactly one terminal \textsc{Respond}.  The default
limits are 20 nodes and dependency depth 12.  \textsc{Observe} and
\textsc{Act} contain a live operation, constant JSON arguments, and a success
key.  \textsc{Ask} and \textsc{Confirm} wait for a correlated user event.
\textsc{Confirm} also names the exact descendant action nodes it authorizes.
\textsc{Respond} has one typed outcome---\texttt{completed},
\texttt{answered}, \texttt{refused}, \texttt{declined}, or
\texttt{fail\_safe}---and an explicit evidence set.

From $T_t$, PACT constructs a canonical, content-addressed capability snapshot.
Each operation's argument schema is checked and executed with a Draft~2020--12
validator \cite{jsonschema2020}; nested objects, arrays, formats, numeric
constraints, combinators, and additional-property rules therefore remain in
the trusted boundary.  Duplicate or malformed capability declarations fail
closed.  If trusted metadata labels an operation as \texttt{observe} or
\texttt{act}, node kind must agree.  PACT does not infer effects from operation
names; absent metadata is represented as \texttt{unknown} and receives no
kind-level guarantee.

\subsection{Receding-Horizon Obligation Execution}

Let $\sigma_t(n)\in\{\textit{pending},\textit{waiting},
\textit{satisfied},\textit{failed}\}$.  The kernel selects the first
topologically ready node
\begin{align}
 R_t={}&\{n\mid\sigma_t(n)=\textit{pending},\notag\\[-1mm]
       &\quad\forall d\in\operatorname{dep}(n),
       \sigma_t(d)=\textit{satisfied}\},\\
 n_t={}&\min_{\prec}R_t.
 \label{eq:ready}
\end{align}
It emits only $n_t$ and then waits.  Immediately before an external call, a
fresh snapshot must have the same digest, contain the operation, preserve any
known effect label, and accept a deep-copied argument object.  Ask and
observation results are semantic data boundaries that trigger a new
short-horizon compile; so does a qualified confirmation that revises scope.
By contrast, an exact unqualified confirmation and a matched \textsc{Act}
result are control events: positive confirmation and action success advance
the already-verified immutable suffix without an LLM call; rejection or
failure terminates deterministically.  A capability change detected while
releasing a pending confirmation triggers replanning without inherited scope;
a change detected at external-call emission fails closed.  Any successful
action key is also placed in $O_t$, so a later horizon cannot silently forget
a partially completed request.

Records carry applicable provenance fields: source and status; event, call, and
node IDs; operation, context, and plan; argument/schema digests; and
authorizations.  Values are isolated through a JSON round trip on write and
read, preventing nested caller mutation from rewriting history.  Fingerprinted
event IDs make identical replay idempotent; conflicting reuse, out-of-order
input, and explicit external failure cannot satisfy that plan.

For runtime events and confirmations, \texttt{schema\_digest} stores SHA-256
of the entire canonical capability snapshot---contracts and trusted
annotations---not merely one operation's parameter schema.

Confirmation is capability-conditional and scoped.  For an operation marked
confirmation-required, the verifier accepts either an ancestor
\textsc{Confirm} authorizing that action node or a prior true confirmation
record matching
\begin{equation}
 (D(T_t),\operatorname{operation},
   \operatorname{SHA256}(\operatorname{arguments})).
 \label{eq:scope}
\end{equation}
Here $D(T_t)$ denotes that full snapshot digest.  After matching action
success, the adapter records the confirmation key, operation, and argument
digest in $C_t$, then projects that scope out of future compiler evidence.  It
cannot authorize another call in a future horizon.  One confirmation record
may explicitly contain several action scopes in the accepted immutable plan;
its other scopes remain independent until individually consumed.

\section{Assurance and Compute Bounds}

We distinguish properties checked by code from assumptions about its inputs.
Suppose the adapter is the only emission path, the live capability declaration
is a valid schema, and hashing/JSON validation behave as specified.

\paragraph{Capability and schema confinement.}
If PACT emits external call $a_t=(o,x)$, then $o\in T_t$ and
$x\models\operatorname{schema}_{T_t}(o)$.  The verifier establishes this for
the accepted immutable plan; the emission guard repeats both membership and
validation on a fresh digest-matched snapshot.  Mutation or revocation between
the two checks yields no call.

\paragraph{Evidence-closed completion.}
For a typed \texttt{completed} response, PlanIR requires at least one action or
carried evidence, the verifier requires every current action-success key and
every key in $O_t$ in the terminal evidence set, and carried keys must be
successful external action records.  At emission, every cited record must be
external with status \texttt{success}.  Hence no failed, user-authored,
missing, or planned event can justify the \texttt{completed} outcome category.
The semantic correctness of response prose remains model-produced.
\texttt{answered} is action-free and cites observations selected by local
lowering; \texttt{refused} calls nothing; \texttt{declined} requires a negative
confirmation.  Emergency
\texttt{fail\_safe} may lack evidence after protocol/compiler failure, but
never claims completion.

\paragraph{Transition confinement.}
Within the kernel, at most one node is waiting.  Only a type-compatible event
for that internal call can change it to satisfied.  External events additionally
match context, plan, node, operation, argument digest, schema digest, and a
nonempty evaluator call ID.  This establishes replay/idempotency properties
\emph{after} the A2A adapter has correlated the sole returned result; the
protocol limitation below is material.

\paragraph{Failure non-escalation (invariant argument).}
Proceed by induction over processed events.  Initially $E_0$ is empty and no
node is satisfied.  Registration of an unmatched event changes only its audit
disposition.  A matched input event can satisfy only the pending
\textsc{Ask}/\textsc{Confirm}.  A matched external event creates success
evidence only when its explicit status is \texttt{success}; failure instead
creates a distinct failure key and marks the node failed.  Duplicate and
conflicting event IDs do not change the first stored record.  Therefore none of
these cases can manufacture a successful action record.  The completion guard
consults only such records, preserving the invariant.

\paragraph{Obligation monotonicity (invariant argument).}
Within one goal, $O_t$ grows after each satisfied \textsc{Act} and is cleared
only at a terminal or explicit safe termination.  Every new compiler request
receives $O_t$.  The verifier rejects a carried key that is not a successful
external action and rejects a completed terminal omitting any member of $O_t$.
Thus, by induction over horizons, a reported completion accounts for all
earlier successful nodes classified as \textsc{Act}.  If a later obligation
fails, the adapter instead emits partial-failure text.

\paragraph{Scope-limited confirmation (invariant argument).}
For an annotated capability, admission requires either an authorizing ancestor
in the same immutable plan or a true ledger record whose schema, operation, and
argument digests match.  After a matching action succeeds, the adapter removes
that individual authorization through $C_t$ from the compiler-visible evidence
projection.  A record may hold multiple action scopes, but each exact scope is
consumed independently; changing any tuple component makes it unusable.

\paragraph{Bounded inference and telemetry.}
For compiler request $j$ within one Track~2 decision step, with repair
indicator $r_j\in\{0,1\}$ and
Act-audit indicator $s_j\in\{0,1\}$,
\begin{equation}
 N_j=1+r_j+s_j\leq3.
 \label{eq:calls}
\end{equation}
The audit is spent only after a locally accepted plan contains an
\textsc{Act}; an action-free answer uses $s_j=0$.  The audited candidate again
crosses the full decoder/verifier boundary.
Input length, conversation count, evidence count, candidate size, graph size,
and graph depth have independent local ceilings.  Provider rate-limit handling
uses a finite retry limit $R_{\max}$ and cumulative wait limit $W_{\max}$,
including proactive pacing and reactive header-aware backoff; exhaustion raises
a typed error rather than waiting indefinitely.  Across tool turns, provider
usage is accumulated until the next text response and exported as A2A
\texttt{turn\_metrics}.  Competition-accounted usage is
\begin{equation}
 B(q)=\sum_{j\in q}(p_j+c^{\mathrm{vis}}_j+h_j),
 \label{eq:tokens}
\end{equation}
where $c^{\mathrm{vis}}_j$ excludes the reasoning tokens $h_j$.  Cerebras
reports reasoning inside its completion total, so the adapter subtracts $h_j$
before populating A2A \texttt{completion\_tokens}; completion and
\texttt{thinking\_tokens} are therefore mutually exclusive.  Call count, mean
call time, model, cost, passes, and quota wait are also reported; safe errors
retain usage from failed verification attempts.  Equation~\ref{eq:tokens} is
evaluator-facing telemetry, not a local hard cap on task-level trajectory
length; PACT instead enforces per-input, per-candidate, per-call, and
per-compilation bounds.

\paragraph{Liveness boundary.}
For a stable accepted DAG, each matched control event changes one waiting node,
so its deterministic suffix reaches a terminal or Ask/Observe boundary within
$|\mathcal N|$ emissions.  A missing user, exhausted provider, malformed
result, or revoked capability can stop progress safely; semantic replanning can
also continue indefinitely.  Bounds therefore apply per compilation and
accepted horizon, not to an open conversation.

\section{Engineering Evaluation and Limits}

We evaluate enforceable mechanisms, not hidden outcomes.  On 17 July 2026,
356 tests plus three subtests covered PlanIR, verification, compilation,
ledger/runtime transitions, A2A, full JSON Schema, and bounded provider retries.
Adversarial cases include cycles, revocation, malformed/replayed results, deep
mutation, confirmation reuse, forgotten actions, exhausted repair, cancellation,
and safe-error metrics.  Generic non-automotive schemas exercise the same
kernel.  These deterministic \texttt{uv run pytest -q} checks do not estimate
benchmark reward or three-trial consistency.

\paragraph{Historical single-task smoke.}
An earlier low/4,096 public \texttt{base\_0} run earned all five subrewards.
Seven calls used 97,922 prompt, 1,940 recomputed visible-completion, and 2,032
thinking tokens (101,894 total), with 360,753.4\,ms quota wait.  It predates the
final audit/control suffix and is not a performance estimate.  A separate
truncation canary motivated the 8,192 ceiling and typed fail-closed handling.

The final closure also passes Ruff and mypy over 18 release files; a local
\texttt{linux/amd64} image serves its A2A Agent Card.  This establishes local
quality and reachability, not public-image validation or live performance.

\paragraph{Operational reproducibility.}
The selective image runs unprivileged on pinned bases; secrets enter only at
runtime, while model/retry settings remain environment-configurable.  Compiler
results expose hashes and value-redacted issues, but raw candidates and the
process-local ledger are not persisted.  Stable context locks prevent a
cancellation replacement-lock race.

Local checks do not prove natural-language interpretation; the second-pass
audit is stochastic.  An operation with \texttt{unknown} effect metadata may
be either \textsc{Observe} or \textsc{Act}, so confirmation guarantees require
an explicit annotation.  A2A also exposes no agent nonce: with one outstanding
call, PACT checks operation and nonempty evaluator ID, but cannot distinguish a
replayed identical result cryptographically.  Hashes are content addresses,
not signatures.

PACT's central result is architectural: stochastic inference proposes meaning,
while a small deterministic layer owns capability admission, evidence-bearing
state changes, and evidence conditions for typed terminal outcomes.  This
separation supports compute-bounded execution without benchmark-specific
workflows and makes both the achieved guarantees and the remaining semantic
uncertainty explicit.
